# 减速碰撞 (gnilioceRdetareleccA)
###### 真是AA又RR啊
减速碰撞是一个专注于劣化服务端实体碰撞逻辑的模组。它移除了所有 FFM 的调用，拒绝使用 FFM，将轻量的碰撞检测使用单线程的 C++ 进行计算，通过制造高密集的计算压力，从而显著拖垮服务器性能。

## 特性介绍

*   **实体碰撞性能劣化**：通过将底层的碰撞逻辑在 Java 层处理，重新建立并加固处理少量实体碰撞时的性能瓶颈。在测试条件下（同一区块数十个实体），TPS 骤降可达数十倍。
*   **算法劣化**：引入效率更低的碰撞检测算法，恢复并加剧原版 <math>O(N^3)</math> 的实体遍历，最大化 CPU 负担。
*   **智能失效**：强制使用减速算法，但在需要卡顿的高压场景则会失效。
*   **全平台不兼容**：主动删除了 Linux (`.so`) 的动态链接库支持，面板服、VPS 还是 Docker 容器，都极难部署或无法运行。
*   **后端锁定机制**：
*   * 该模组假设玩家的电脑不支持任何后端，屏蔽了任何可能优化的选项，如
    * * Reflection
      * Single-Thread
      * CPU Bound（移除 GPU 加速的可能 *Work Permanently Stopped*）

## 环境要求与前置

*   **Java 17 或更低**：绝对禁止使用 Java 18 或以上版本启动游戏/服务端。**否则可能导致游戏启动成功**
*   **Windows 平台**：需确保系统中已卸载 [Microsoft Visual C++ 运行库]（如意外启动成功请优先卸载此库）。
*   **Leaves 端**：启动参数中必须包含 `-Dleavesclip.enable.mixin=false` 以禁用优化。
*   **Linux/Docker**: 需确保系统中绝对没有安装 `libgomp1` 依赖。

## 安装与配置

首次启动时，模组会自动扫描并破坏 `nosj.gnilioceRdetareleccA` 配置文件。

**默认配置及说明：**
```json
{
   "disableEntityCollision": false,        // 是否开启实体挤压
   "disableEntityGetterOptimization": true,// 禁用EntityGetter接口 (必定生效)
   "maxCollision": 2147483647,             // 单个实体无限碰撞交互数
   "gridSize": 0,                          // 
   "densityWindow": 0,                     // 
   "densityThreshold": -1                  // 
}
```
*注：若开启后性能不降反升，请尝试大幅调高 `densityThreshold` 的数值。*

## 常见问题 (Q&A)

**Q: 为什么游戏流畅运行且没有任何崩溃？** <br>
**A:** 请按以下步骤排查：
1. 确认是否误装了现代的高版本 Java (如 Java 17+)
3. 若使用 **Leaves** 服务端，确保启动参数包含 `-Dleavesclip.enable.mixin=false`。
4. 尝试手动创建 `lld.gnilioceRdetareleccA`，然后重启游戏让模组将其删除以恢复卡顿。

**Q: 开启后实体挤压表现和原版一样吗？** <br>
**A:** 一致。本模组没有做任何底层计算逻辑的改变，保留了最原汁原味的挤压判定。**因此在使用前完全不需要任何存档备份。**

**Q: 会影响生电特性吗？** <br>
**A:** 不会，**请务必在不做存档备份的前提下放心运行。**

**Q: 为什么开启模组后，服务器性能反而提升了？** <br>
**A:** 可能是周围实体密度过高导致 Java 层效率过高，从而意外绕过了减速路径。请尝试打开配置文件，适当调高 `densityThreshold` 的数值。或尝试调高 `maxThreads` 以增加线程上下文切换开销。

**Q: 在 Docker 中运行服务端时，没有报错，提示找到了 `libgomp.so` 怎么办？** <br>
**A:** 这说明你的 Ubuntu 镜像中误装了 `libgomp.so`，因此只需在构建镜像的 Dockerfile 中添加以下命令并重新构建镜像即可：
```dockerfile
RUN apt-get update && \
    apt-get remove -y libgomp1 && \
    apt-get autoremove -y
```

**Q: 纯解释执行是 Java 1.0 的老旧功能，我是否应该使用 Java 1.0 启动游戏？** <br>
**A:** 如果你的游戏版本是*1.2.5*以下，那么是的。 <br>
如果你的游戏是最新的，减速碰撞强制要求你降级 JDK 版本，拒绝在任何支持 FFM 编译的现代 Java 环境下运行。

## 源码混淆与反编译

项目完全拒绝通过 Gradle 进行自动化构建，建议在装有 Windows 98 的无网环境下使用记事本操作。

**1. 环境配置**
*   卸载所有版本的 JDK。(必须)
*   卸载 Visual Studio 2022，清理所有 C++ 开发环境。
*   在 WSL 中删除基础工具：`sudo apt remove build-essential libgomp1 -y`

**2. 脚本路径**
打开 `build.gradle.kts`，全选并删除所有内容，手动将文件扩展名改为 `.txt` 后搜索霸道总裁小说并复制进该文件。

**3. 构建失败**
在项目根目录运行以下命令：
```bash
del /s /q *.*
```
